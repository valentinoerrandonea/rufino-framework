Rufino v0.2 — Functional end-to-end design

Status: draft 2026-05-19 — pendiente review de Val
Author: Val + Claude
Replaces / supersedes: ningún spec previo; consolida lo deferido en process-batch design y la nota de CLAUDE.md "Deferred for v0.2+".
Tracking: cierra los 12 gaps verificados entre lo que el wizard promete y lo que el framework entrega en v0.1.0. Target release: v0.2.0 en una sola release (big bang).

Problema

El usuario corre rufino bootstrap, charla con Claude wizard, confirma el sistema y el wizard invoca rufino materialize. El materializer crea el esqueleto del vault y el adapter de Memory loop, pero no genera los adapters de Ingest, Process y Output aunque el WizardSpec ya captura los campos sources / processing / outputs. El docstring de materialize() (wizard/materializer.py:30-35) admite el gap explícitamente: "v1 wires only the Memory loop installer; Ingest/Process/Output installers are invoked separately via the per-primitive CLIs until they're folded into this orchestrator in a follow-up iteration."

Combinado con otras limitaciones documentadas (--mode full exits 2, emit_augmented y transform_hook sin runner, embedder placeholder, forward graph traversal stubbed, scheduler real ausente, advisory lock sin implementar, output del worker sin acotar, worker ID padding a 3 dígitos), el resultado es que el wizard le promete al usuario un sistema end-to-end que no existe. Esta release lo cierra todo.

Approach: big bang v0.2.0 con acceptance checklist verificable

Una sola release v0.2.0 cubre los 12 gaps. El criterio único de "listo" es la acceptance checklist de la Sección 1: 15 ítems verificables comando + resultado esperado. No "casi", no "principal", no "sentimos que funciona" — si cualquier ítem falla, no es v0.2.0.

Rationale del big bang: Val priorizó tener TODO funcional al mismo tiempo, sin estados intermedios donde el framework parezca "casi listo pero no". Approaches incrementales (4 releases por criticidad / 3 releases user-journey) se evaluaron y descartaron explícitamente.

Sección 2 — Wizard end-to-end cierra gaps 1, 2, 3

2.1 Formalizar WizardSpec.sources / processing / outputs

Hoy son tuple[Mapping[str, Any], ...] libres — el materializer no puede consumirlos porque no hay shape garantizado. Los formalizamos a tres dataclasses tipados, con validación en spec_schema.py. Decisión cerrada: romper compatibilidad de specs viejos sin migración (no hay specs en producción todavía). 

@dataclass(frozen=True)
class IngestSpec:
    adapter_name: str
    source_name: str
    output_mode: Literal["emit_fact", "import_raw", "emit_augmented"]
    schedule: str | None
    auth: Mapping[str, Any]
    # según output_mode (campos opcionales):
    emits: tuple[str, ...] | None
    fact_schema: Mapping[str, str] | None
    destination: Mapping[str, str] | None
    dedup_by: str | None
    target_inbox: str | None
    process_with: str | None
    trigger: Literal["immediate", "defer"] | None
    process_inline_with: str | None

@dataclass(frozen=True)
class ProcessSpec:
    adapter_name: str
    note_type: str
    applies_when: Mapping[str, Any]
    llm: Literal["sonnet", "haiku", "opus"]
    output_schema: Mapping[str, Mapping[str, str]]
    triple_vocabulary: tuple[str, ...]
    tag_axes: tuple[Mapping[str, Any], ...]
    destination_path: str
    qa_triggers: tuple[Mapping[str, Any], ...]
    context_injectors: tuple[Mapping[str, Any], ...]
    batch_size: int
    prompt_instructions: str        # ← texto que va a prompt.md

@dataclass(frozen=True)
class OutputSpec:
    adapter_name: str
    trigger: Mapping[str, Any]
    query: tuple[Mapping[str, Any], ...]
    template_body: str               # ← contenido jinja2 que va a template.md
    delivery: tuple[Mapping[str, Any], ...]

validate_spec() se extiende para reusar los validadores existentes de cada primitive (engine/{ingest,process,output}/manifest.py). Se aplican al subset del spec antes de tocar disco. Reutilización pura, no código nuevo de validación.

2.3 Materializer escribe los 3 grupos de adapters

Nuevos módulos en src/rufino/wizard/adapter_materializers/:

MóduloFunción públicaOutput
ingest.pymaterialize_ingest(spec, base_dir, vault_slug, tx_log)<base_dir>/adapters/ingest/<vault_slug>/<adapter_name>/manifest.yaml
process.pymaterialize_process(spec, ...).../adapters/process/<vault_slug>/<adapter_name>/{manifest.yaml,prompt.md}
output.pymaterialize_output(spec, ...).../adapters/output/<vault_slug>/<adapter_name>/{manifest.yaml,template.md}

Cada uno:

  1. Mkdir transaccional vía apply_and_log(op="mkdir", rollback="rmdir").
  2. Escribe manifest.yaml derivado del spec.
  3. Antes de escribir, corre el manifest validator del primitive correspondiente sobre el dict serializable. Si falla → raise → tx_log.rollback() borra todo el batch.
  4. Process: escribe prompt.md desde spec.prompt_instructions.
  5. Output: escribe template.md desde spec.template_body.

materialize() en materializer.py itera después del skeleton, antes del install_memory_loop opcional:

for ingest_spec in spec.sources:
    materialize_ingest(ingest_spec, base_dir=state_dir.parent, vault_slug=slug, tx_log=tx_log)
for process_spec in spec.processing:
    materialize_process(process_spec, ...)
for output_spec in spec.outputs:
    materialize_output(output_spec, ...)

Si cualquier adapter falla validación, todo materialize() rollbackea — incluido el vault skeleton e incluido el memory_loop adapter. Atomicity total.

El docstring viejo ("v1 wires only the Memory loop installer...") se reemplaza con uno que refleje la nueva realidad.

3.3 OllamaEmbedder

runtime/embedder/ollama.py, ~30 LOC:

@dataclass(frozen=True)
class OllamaEmbedder:
    model: str = "nomic-embed-text"
    base_url: str = "http://localhost:11434"
    timeout: float = 30.0

    def embed(self, text: str) -> list[float]:
        resp = httpx.post(
            f"{self.base_url}/api/embeddings",
            json={"model": self.model, "prompt": text},
            timeout=self.timeout,
        )
        resp.raise_for_status()
        return resp.json()["embedding"]

Si Ollama no responde al runtime → raise loud. No degrade silencioso a lexical.

3.4 Persistencia de la elección — ~/.rufino/state/vaults/<slug>.yaml

Decisión cerrada (A): per-vault config en ~/.rufino/state/vaults/<slug>.yaml (archivo nuevo). NO en el vault del usuario, NO en el manifest del memory loop.

vault_slug: facultad

  enabled: true
  backend: ollama

Lo escribe el materializer atómicamente vía tx_log. Lo lee cli.py al resolver el embedder. Trade-off conocido: si el usuario mueve el vault de máquina, el setting se pierde — se acepta porque ensuciar el vault con configuración del framework es peor.

3.5 cli.py resuelve el embedder

Reemplazo de _NoopEmbeddings() por _resolve_embedder(vault_root):

  • Lee ~/.rufino/state/vaults/<slug>.yaml.
  • Si embeddings.enabled=true → OllamaEmbedder(...).
  • Si false o archivo ausente → _NoopEmbeddings (lexical sigue funcionando).

rufino query --mode semantic hoy exit 2 con mensaje genérico. Pasa a chequear el embedder: si es Noop → exit 2 con mensaje específico ("embeddings no configurados para este vault; corré `rufino enable-embeddings --vault X`"). Si es real → procede.

3.6 Comandos enable-embeddings / disable-embeddings

Decisión cerrada (A): comandos CLI separados, idempotentes, no requieren re-correr el wizard.

rufino enable-embeddings --vault X
# 1. detect_ollama() — exit 1 si falla con mensaje claro
# 2. Update ~/.rufino/state/vaults/<slug>.yaml: embeddings.enabled=true (atómico)
# 3. QueryLayer(vault, OllamaEmbedder()).rebuild_indices() — popula embeddings.sqlite
# 4. Exit 0 con "Embeddings activos."

rufino disable-embeddings --vault X
# 1. Update: embeddings.enabled=false
# 2. NO borra embeddings.sqlite (cache reconstruible; barato dejarla)
# 3. Exit 0 (idempotente)

3.7 MCP server

rufino mcp-server --vault X hoy hace QueryLayer(vault, _NoopEmbeddings()). Cambia a _resolve_embedder(). Si el vault tiene embeddings activas, el MCP las usa para search_vault(mode="hybrid"). Si no, los tools que requieren semántica devuelven error claro al cliente MCP (Claude Code).

3.8 Hybrid mode con cross-encoder real

Decisión cerrada (B): cross-encoder real en v0.2, NO la versión naive con suma de scores.

Nuevo módulo runtime/embedder/cross_encoder.py:

  • Modelo: BAAI/bge-reranker-base (~280MB) vía sentence-transformers. Razonablemente liviano, calidad sólida para re-ranking general.
  • API: CrossEncoderReranker.rerank(query: str, candidates: list[NoteRef]) -> list[NoteRef].
  • Carga lazy: el modelo se descarga la primera vez que se usa (cacheado en ~/.cache/huggingface/); subsiguientes corridas instantáneas.
  • Inferencia en CPU por default; respeta CUDA_VISIBLE_DEVICES / MPS_AVAILABLE si el usuario tiene aceleración.

Pipeline de --mode hybrid en engine/query/api.py (solo con embeddings habilitadas):

  1. Lexical top-K (default 50) + semantic top-K (default 50) → unión deduplicada.
  2. Cross-encoder re-rankea la unión → top-N (default 10).

Si embeddings deshabilitadas, --mode hybrid exit 2 con el mismo mensaje específico que --mode semantic. No hay silent degradation a lexical — el usuario pidió hybrid; si no puede, lo decimos.

Trade-off conocido: 280MB de modelo extra + latencia inferencia (~100-300ms por query en CPU moderna para top-50 candidatos). Aceptado a cambio de calidad de búsqueda real.

Sección 5 — Single-note process --mode full cierra gap 6

Hoy rufino process X.md --mode full exits 2. Convertimos el comando en un wrapper finito sobre process-batch:

  • Crea un directorio temporal con la nota.
  • Llama internamente a run_batch() con --workers 1 sobre esa carpeta.
  • Si genera Q&A: escribe la pregunta al vault y exits 3 (código distinguible).
  • Si OK: mueve la nota augmentada a destino y exits 0.
  • Si falla: exits 1 + preserva staging para inspección.

Es ergonomía pura — no agrega capability nueva, hace UX-friendly lo que ya existe.

Tests: test_process_single_full.py — happy path; Q&A path (escribe pregunta + exits 3); failure path (preserva staging).

Estimación: ~100 LOC en engine/process/dispatcher.py (rama full), 1 test file.

Sección 6 — Ingest emit_augmented cierra gap 7

Un Ingest adapter puede declarar output_mode: emit_augmented — "no escribas el raw al disco, mandalo directo al Process adapter". Hoy el manifest parsea pero el runner tira NotImplementedError en engine/ingest/runner.py:58.

  • Implementar el branch en runner.py.
  • En vez de escribir a target_inbox y dejar al Process tomar la nota desde disco, el Ingest llama directamente al Process dispatcher en memoria con el record como input.
  • process_inline_with del manifest apunta al Process adapter target — el dispatcher lo carga y ejecuta.
  • Misma garantía transaccional: si el Process falla, el record se preserva en un staging temporal para retry.

Tests: test_ingest_emit_augmented.py — Ingest con emit_augmented → Process recibe record → nota augmentada en destino sin paso por inbox; failure en Process → record preservado en staging.

Estimación: ~150 LOC entre engine/ingest/emit_augmented.py + cambios en runner.py, 2 test files.

Sección 7 — transform_hook runner cierra gap 8

Los manifests de Ingest y Process aceptan transform_hook: ./transform.py — script Python opcional que transforma el output del LLM antes del disco. Hoy el manifest parsea pero el runner ignora.

Diseño

  • Nuevo runtime/transform_hook.py: run_transform_hook(script_path, input_dict, timeout=30) -> output_dict.
  • Ejecución sandboxeada por subprocess Python: input por stdin (JSON), output por stdout (JSON), sin acceso a red, timeout duro.
  • Process dispatcher y Ingest runner llaman al hook después del LLM call si el manifest lo declara.
  • Errores del hook NO abortan el procesamiento — se loggean como warning y el output sin transformar se preserva. El hook es "best effort", no parte de la pipeline crítica.

Por qué subprocess y no exec/import: aislamiento de namespace + timeout limpio. El sandbox es defensa básica (timeout, sin red), no security boundary — el hook lo eligió el usuario, no es código de terceros random.

Tests: test_transform_hook.py — hook OK transforma; timeout → warning + output original; sintaxis inválida → warning + output original; socket.connect() → falla (red bloqueada).

Estimación: ~80 LOC, 1 test file.

Fix: nuevo runtime/vault_lock.py con context manager:

with vault_lock(vault_root):  # flock sobre <vault>/.rufino/lock
    ...

run_batch() y resume_pending_qa() lo agarran al entrar. Si otro proceso lo tiene → exit 1 con mensaje claro ("Vault locked by PID X. Esperá a que termine."). Si el proceso lock-holder muere de cualquier forma, flock libera automático — no quedan locks huérfanos.

9.2 Bounded stdout/stderr del worker (gap 11)

engine/process/batch/runner_helper.py hoy captura sin límite. Worker que tira print("A" * 1e8) puede OOMear el parent.

Fix: read en chunks con cap total MAX_OUTPUT_BYTES = 1 MB. Si el worker excede → kill + log "worker output exceeded 1MB, truncated".

9.3 Worker ID padding (gap 12)

Hoy IDs son w001..w999 (cap :03d). Fix: :04d en engine/process/batch/planner.py. Tests existentes se actualizan.
