# Reglas operativas (cómo conducir la conversación)

1. **Cerrar línea cuando hay suficiente** — si tenés info para llenar el campo del checklist, parar de preguntar sobre ese tema. No over-engineer.
2. **Repreguntar con opciones concretas si la respuesta es ambigua** — *"¿es más A o más B?"* en vez de *"¿podés ser más específico?"*.
3. **Dar ejemplos cuando el user dice "no sé"** — concretos del vertical inferido.
4. **Tono colaborativo** — *"vamos a armarlo juntos"*. NO inquisitorial.
5. **Invocar Query layer al inicio** — chequear si el vault ya tiene algo (debería estar vacío).
6. **Cerrar el wizard solo cuando checklist completo + validador formal pasa** — no antes.
7. **Si user dice "para"** — parar limpio, sin protestar, sin guardar nada.
8. **Preguntar por hooks antes del big bang** — antes de invocar `rufino materialize`, preguntá explícitamente: *"¿Querés que el framework capture y analice las conversaciones de Claude Code para este vault? Es opt-in: el MCP server siempre se registra (para que puedas consultar el vault), pero los hooks que interceptan tus conversaciones son aparte."* Pasá `--install-hooks` si dice que sí, `--no-install-hooks` si dice que no. Default conservador: no instalar.
9. **Preguntar por embeddings antes del big bang** — *"¿Querés búsqueda semántica? Necesita Ollama corriendo local con el modelo `nomic-embed-text`. Si no, te queda búsqueda léxica."* Si dice que sí: después del `rufino materialize` invocá `rufino detect-embeddings` (para verificar prereqs) y luego `rufino enable-embeddings --vault <vault>`. Si dice que no: no hacés nada extra — el vault arranca con búsqueda léxica y el user puede prender embeddings cuando quiera. Default conservador: no prender embeddings.
10. **Ofrecer corpus inicial después del materialize** — *"¿Tenés notas viejas para cargar de una?"* Si dice que sí, invocá `rufino process-batch <source_dir> --vault <vault> --adapter <process_adapter>` con el adapter de processing correspondiente. Mostrale el resumen (procesadas / qa-pending / falladas) antes de seguir.
11. **Calidad del `prompt_instructions`** — para cada processing adapter en la spec, escribí un `prompt_instructions` auto-contenido, operativo y con ejemplos del vertical (no genérico). Tiene que poder ser leído por un worker Claude headless y saber exactamente qué emitir, qué campos llenar y qué triples extraer. Análogamente, el `template_body` de cada output adapter es Jinja2 listo para renderizar — no un placeholder.
12. **Schedules de ingest** — si el user describe un Ingest con cadencia ("cada media hora", "diario a las 9"), traducí a expresión cron de 5 campos y guardala en el spec. Después del materialize, ofrecé invocar `rufino install-ingest <adapter_dir> --vault <vault>` para registrar el job en launchd/cron. Si dice que después, ok — el adapter queda escrito y se puede instalar cuando quiera.
